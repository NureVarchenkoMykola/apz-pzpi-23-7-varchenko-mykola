2 ЗАВДАННЯ
     Метою практичного заняття є ознайомлення з одним із шаблонів проєктування програмного забезпечення, дослідження його призначення, архітектурної структури, ролей основних учасників та можливостей практичного застосування під час розроблення програмних систем.
     У цій роботі розглядається поведінковий шаблон проєктування State, який використовується для організації поведінки об’єкта залежно від його внутрішнього стану. Потрібно дослідити особливості побудови цього патерна, принцип взаємодії між компонентами, а також визначити випадки, у яких його використання є найбільш ефективним.
     У процесі виконання роботи необхідно підготувати доповідь і презентаційні матеріали, створити приклади програмної реалізації патерна State та пояснити їх у межах відеодоповіді. Також потрібно виконати аналіз переваг і недоліків шаблону, розглянути типові сфери його застосування та показати відмінності від альтернативних підходів до керування станами об’єктів.


3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про шаблони (патерни) проєктування
     Шаблони проєктування програмного забезпечення - це типові архітектурні рішення, які використовуються для розв’язання поширених задач під час створення програмних систем. Вони описують перевірені підходи до побудови структури програм та взаємодії між їх компонентами.
     Основною метою використання патернів є підвищення якості програмного забезпечення, спрощення підтримки коду, зменшення складності системи та забезпечення можливості повторного використання архітектурних рішень.
     Найбільш відомою класифікацією шаблонів проєктування є класифікація GoF (Gang of Four), відповідно до якої всі патерни поділяються на три групи:
• породжувальні;
• структурні;
• поведінкові.
     Породжувальні шаблони відповідають за створення об’єктів та приховують складність процесу їх ініціалізації. Структурні патерни визначають способи побудови складних структур із класів та об’єктів. Поведінкові шаблони описують взаємодію між об’єктами та організацію їхньої поведінки.
     Патерни проєктування широко використовуються під час створення вебзастосунків, мобільних програм, інформаційних систем, ігрових рушіїв та інших програмних продуктів.
     3.2 Опис обраного патерна
Назва патерна: State
Тип шаблона (патерна): поведінковий
     Основне призначення:
     Патерн State призначений для зміни поведінки об’єкта залежно від його внутрішнього стану. Шаблон дозволяє винести логіку різних станів в окремі класи, що зменшує кількість умовних операторів та робить код більш структурованим.
     Передумови застосування:
     Використання патерна State є доцільним у випадках коли поведінка об’єкта залежить від його поточного стану, у програмі присутня велика кількість умов if або switch, система має декілька режимів роботи, або необхідно спростити підтримку та розширення програмного коду.
     Переваги:
• покращення читабельності коду;
• зменшення кількості умовних операторів;
• спрощення розширення функціональності;
• ізоляція логіки окремих станів;
• відповідність принципу відкритості/закритості.
     Недоліки:
• збільшення кількості класів у програмі;
• ускладнення структури невеликих проєктів;
• необхідність створення окремого класу для кожного стану.
3.3 Структура шаблона (патерна)
     Патерн State складається з декількох основних компонентів.
• Context
     Основний клас, який зберігає поточний стан об’єкта та делегує виконання дій відповідному стану.
• State
     Базовий інтерфейс або абстрактний клас, який визначає спільний набір методів для всіх станів.
• ConcreteState
     Конкретні реалізації станів, які містять власну логіку роботи та можуть змінювати поточний стан об’єкта Context.
     Опис взаємодії об’єктів:
     Клас Context містить посилання на поточний стан та передає йому виконання певних дій. Конкретний стан реалізує необхідну поведінку та може змінювати стан Context на інший. Завдяки цьому поведінка системи змінюється динамічно під час виконання програми.

     UML-діаграма з поясненням:

Рисунок.3.1 - UML-діаграми патерна State

     На UML-діаграмі (див.рис.3.1) показано структуру патерна State. Клас Context містить посилання на поточний об’єкт стану StateBase та делегує йому виконання дій.
     StateBase є базовим інтерфейсом або абстрактним класом для всіх станів. Класи State A, State B та State C реалізують конкретну поведінку системи залежно від поточного стану об’єкта.

     3.4 Опис реалізації мовою програмування C#
     Для демонстрації роботи патерна State було реалізовано приклад моделі медіаплеєра мовою програмування C#. У розробленій системі поведінка об’єкта змінюється залежно від його поточного стану. Такий підхід дозволяє уникнути використання великої кількості умовних операторів та спрощує структуру програмного коду.
     У прикладі система може перебувати у трьох основних станах:
• ReadyState - стан готовності до роботи;
• PlayingState - стан відтворення музики;
• LockedState - заблокований стан.
     Кожен стан реалізується окремим класом, який містить власну логіку поведінки та може змінювати поточний стан об’єкта Context.
     Основні етапи реалізації:
     1) Створення базового інтерфейсу (див.додаток.в)
     На першому етапі було створено інтерфейс IState, який визначає спільний метод Handle для всіх можливих станів системи. Наявність єдиного інтерфейсу забезпечує однаковий механізм взаємодії між класом Context та конкретними станами.
     Інтерфейс дозволяє реалізувати поліморфізм та спрощує додавання нових станів без зміни існуючого коду програми.
     2) Реалізація класу Context (див.додаток.в)
     Клас Context є основним елементом патерна та зберігає посилання на поточний стан об’єкта. У ньому реалізовано метод SetState, який дозволяє змінювати активний стан системи.
     Також Context містить метод Request, який передає виконання дій поточному стану. Завдяки цьому основний клас не містить складної логіки перевірок і делегує виконання поведінки конкретним станам.
     3) Реалізація конкретних класів станів (див.додаток.в)
     Для кожного стану створено окремий клас:
• ReadyState;
• PlayingState;
• LockedState.
   Кожен клас реалізує інтерфейс IState та містить власну реалізацію методу Handle. Наприклад, у стані ReadyState система переходить до стану PlayingState, а у стані PlayingState до LockedState. Це демонструє механізм динамічної зміни поведінки об’єкта під час виконання програми.
   Такий підхід дозволяє ізолювати логіку окремих станів та спростити подальше розширення функціональності системи.
     4) Демонстрація використання
     Під час виконання програми об’єкт Context змінює свою поведінку залежно від поточного стану. Завдяки використанню патерна State вдається уникнути великої кількості умовних операторів та зробити структуру програми більш гнучкою й масштабованою.


4 ВИСНОВКИ
     У процесі виконання практичної роботи було досліджено поведінковий шаблон проєктування State. Було розглянуто його призначення, особливості структури, принцип роботи та основні сфери застосування.
     У ході роботи було встановлено, що патерн State дозволяє організувати зміну поведінки об’єкта залежно від його внутрішнього стану без використання великої кількості умовних операторів. Завдяки винесенню логіки станів в окремі класи покращується читабельність коду, спрощується підтримка та розширення програмної системи.
     Також було виконано практичну реалізацію патерна мовою програмування C#. У прикладі продемонстровано роботу об’єкта в різних станах та механізм переходу між ними.
     Отже, шаблон проєктування State є ефективним інструментом для побудови гнучких програмних систем із великою кількістю станів та складною логікою поведінки.
